# CareerOps

A Docker-backed, single-user command center for running a focused job search:
track companies and job leads, applications and their follow-ups, interviews, and
AI-assisted fit analysis — all in one place instead of a sprawling spreadsheet.

## Why it exists

A serious job search produces dozens of leads, each with its own status, deadlines,
contacts, and next actions. Spreadsheets lose that thread quickly. CareerOps keeps the
whole pipeline visible, surfaces what needs attention today, and (from Phase 6) uses AI
to analyze job fit and prepare for interviews. It is built as a personal tool first and a
clean-engineering portfolio piece second.

## Tech stack

- **Backend:** ASP.NET Core (.NET 10) Minimal APIs, Clean Architecture + pragmatic DDD,
  EF Core 10 (code-first) on PostgreSQL, Mapster, FluentValidation, Serilog,
  RFC 7807 `ProblemDetails`. OpenAPI via the built-in `Microsoft.AspNetCore.OpenApi`
  (`/openapi/v1.json`) with a Scalar UI at `/scalar/v1`.
- **Frontend:** React 19 + Vite + TypeScript, TanStack Query, React Hook Form + Zod,
  Tailwind CSS + shadcn/ui, react-router, date-fns. The typed API client is generated by
  **orval** from the runtime OpenAPI document — never hand-written.
- **Runtime:** Docker Compose (PostgreSQL + API). The frontend runs on the host via Vite.
- **Tooling:** `just` command runner; `dotnet` and `npm` CLIs for all scaffolding,
  dependencies, and migrations.

## Run with Docker

Prerequisites: Docker Desktop, .NET 10 SDK, Node 22+ LTS, and `just`
(`winget install Casey.Just` / `scoop install just` / `choco install just`).

```sh
cp .env.example .env   # one-time; safe local defaults
just up                # builds and starts PostgreSQL + the API
```

The API listens on `http://localhost:8080` (health at `/health`, OpenAPI at
`/openapi/v1.json`, Scalar UI at `/scalar/v1`).

## Run the backend locally (inner loop)

```sh
just up    # ensure PostgreSQL is running
just api   # dotnet watch on the host, hot-reloading the API
```

## Run the frontend locally

```sh
just web   # Vite dev server on http://localhost:5280
```

> The frontend runs on the host via Vite **by design** (Decision D10); it is intentionally
> not a Docker service. The backend and PostgreSQL run via Docker Compose. This keeps the
> frontend self-contained and able to move to its own repository unchanged.

## Environment variables

Copy `.env.example` to `.env` and adjust as needed. Never commit `.env`.

| Variable | Purpose |
|----------|---------|
| `POSTGRES_DB` / `POSTGRES_USER` / `POSTGRES_PASSWORD` | PostgreSQL container credentials |
| `ASPNETCORE_ENVIRONMENT` | `Development` for the local workflow |
| `VITE_API_BASE_URL` | Frontend → API base URL (`http://localhost:8080`) |

The **DB connection string is not in `.env`** — it is set per run context: the host loop
(`just api`) uses `appsettings.Development.json` (`Host=localhost`), and the container
(`just up`) uses the value in `docker-compose.yml` (`Host=careerops-postgres`). See D21.

## Development workflow

Three loops plus two gates:

- `just up` / `just down` — full stack in Docker (Postgres + API).
- `just api` — backend inner loop (`dotnet watch` on the host).
- `just web` — frontend inner loop (Vite on the host).
- `just gen-client` — regenerate the typed API client from the OpenAPI document (orval).
- `just verify` — the CI-like local gate: backend build + tests, frontend typecheck + build.

Work is delivered in **thin vertical slices** (entity → migration → endpoint → generated
client → page), each ending runnable and demoable. See `docs/knowledge-base/` for the
architecture, conventions, decisions, and the delivery plan.

## Known limitations

- **Single-user, no authentication** — personal-use tool by design.
- **No in-app AI** — analysis runs in external AI agents via the MCP server; no API key required.
- No multi-user, scraping, calendar/email integration, or file upload (see PRD §7.3, §19.1).
